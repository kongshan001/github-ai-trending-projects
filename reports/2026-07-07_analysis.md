## 一、Superpowers 是什么:给 Coding Agent 的完整 SDLC 方法论

**官方一句话定位**: "An agentic skills framework & software development methodology that works." — 一套完整的软件开发方法学,基于一组可组合的 skills,让你现有的 Coding Agent 立刻拥有 SDLC 纪律。

**支持的 IDE/CLI**(README 列出的 install 渠道):
- **Claude Code** / **Antigravity** (agy) / **Codex App & CLI** / **Cursor** / **Factory Droid** / **GitHub Copilot CLI** / **Kimi Code** / **OpenCode** / **Pi**

**它不是**:
- ❌ 不是一个新 IDE
- ❌ 不是替代 Claude/Codex/Gemini 的模型
- ❌ 不是 Mac/Windows 上的图形化应用

**它是**:
- ✅ 通过 plugin/skills 注入的 **方法论层**(位于模型之上、IDE 之下)
- ✅ 在 subagent-driven-development(SDD)流程上做"指挥与编排"

## 三、Superpowers 的 7 个典型动作

**视频里的解构**(对没看过的人来说),Superpowers 典型的工作流由这 7 步构成:

1. **Brainstorming(脑暴)** — Agent 不直接冲进去写代码,先反问目标、边界、风险、验收方式,产出一份设计文档
2. **Worktree 隔离** — 用 Git worktree 检查/创建隔离工作区(每个任务独立分支)
3. **Plan 编写** — Plan 写到文件级别,包含文件路径、接口、测试命令与预期结果,每个任务可独立测试 + 独立审查,子任务级别是 2-5 分钟的动作
4. **TDD 实现(SDD: Subagent-Driven Development)** — Controller 给每个任务派一个 Subagent,Subagent 只拿自己需要的上下文,严格 Red-Green-Refactor 循环
5. **双面 Review** — 同一个 Reviewer 子 agent 在一轮审查里同时出两个 verdict:Spec Compliance + Code Quality
6. **Whole Branch Review** — 所有任务结束后,做一次整分支审查(用最强模型,而不是 task-by-task 重审)
7. **Finishing a Development Branch** — 跑全量测试,选择合并 / 提 PR / 保留分支 / 丢弃

## 四、Superpowers v6 真正的工程突破(三大改动)

视频说的"v6 的真正优化"对应博客中 Jesse Vincent 公开的 36 小时 / 25 个实验的优化记录,以及 v6.0.0 release notes 的工程改动。以下按"贡献度从大到小"排列:

### 4.1 Reviewer Handoff 改:脚本化确定性信息(-10% token & wall-clock)

**问题**(Fable 在分析数千 SDD session 后发现):
- Code Reviewer 和 Spec Compliance Reviewer 在审查时常常**自己跑大量 git 命令**(git log/diff/status)
- 这些命令输出不是"白来的",而是会占满 observation,后续每轮对话反复携带
- 中等规模重构轻松吃掉**几万 tokens**
- 这份信息其实**前面已经被 Coordinator 拆任务看过、Implementer 改过**,Reviewer 又用 git 重新曲变了

**v6 解法**:**任务交接链改为文件,而不是粘贴文本**

```
Reviewer 启动前
   ↓
脚本生成 review package (commit list + diff stat + 完整 diff + task brief)
   ↓
Reviewer 读 review package 文件,不需默认自己跑 git
```

**收益**: 单独这一改动就带来 **~10% 的 token 和 wall-clock 改善**

**类似思路也用在**: Subagent 不再让 Controller 把整份 Plan 反复塞给它,而是把当前任务抽成一个 **task brief 文件**,Subagent 读自己的 brief,不需在 Plan 里翻找任务。

> **工程哲学**: 能用脚本稳定算出来的信息,就别让 LLM 用上下文去查。

### 4.2 架构合并:两个 Reviewer → 一个 Task Reviewer(-15% token + 更严格)

**v5 的设计**(听起来很安全):
- 每个任务有 **2 个独立 Reviewer**:Spec Compliance + Code Quality
- 两边都过才进下一步

**v5 的问题**(实验证明):
- **贵**:每个任务同时跑 2 个独立 reviewer
- **易被博弈**:reviewer 的提示方式影响判定(廉价的提示方式被博弈后失效)
- **冗余**:两个 reviewer 经常重复读同一份 diff

**v6 的合并**: `task-reviewer-prompt.md`(单一文件)

```
一个 Reviewer 读一次 diff
   ↓
同时返回两个 verdict:Spec Compliance + Code Quality
   ↓
一个 fix pass 同时清理两个问题
```

**新增的关键状态: `can-not-verify-from-diff`**

- 有些需求不在当前 diff 里,或者横跨多个任务
- v5 时 Reviewer 经常到处 grep/cat 试图自己补全"全局视角",**既吃 token 又破坏任务边界**
- v6 让 Reviewer 承认自己看不到,标成警告,交给 Controller 处理
- Controller 有 Plan + 跨任务上下文,更适合做这个判断

> **关键的反直觉发现**(evals 暴露):只给 Reviewer 审 diff package、**不给 task brief** 是个陷阱
> - Reviewer 会**很自信地**给 spec verdict,但悄悄地**把 "spec" 重新定义成 "global constraints"**
> - 5 次里 0 次标记出 missing brief
> - 这是个未被发现的失败模式,提示:**省 token 不是粗暴删上下文,要保留任务定义 + 接口 + 测试 + 全局约束**

### 4.3 模型分层 + 必须显式声明 model

**v6 强制规则**: 每次 dispatch **必须**写明使用的 model

**为什么**:
- 不写明的话,**很多 Harness 会默认继承当前 session 里最贵的模型**
- release notes 提到:"有一次 26 个 Reviewer 全部跑在最高模型上" — Token 是这样悄悄烧掉的

**推荐路由**:
| 任务形态 | 推荐模型 |
|---|---|
| Implementer 在 Plan 给出完整代码后转写 | 便宜模型(可降级) |
| Implementer 跑测试、修小错 | 便宜模型 |
| Reviewer 面对小 diff 机械改动 | 便宜模型 |
| **Coordinator 读 Plan + 多步推理 + 架构判断** | **贵模型**(Opus) |
| **Reviewer 设计并发/工程状态/跨文件契约** | **贵模型** |
| Whole Branch Review | **最强模型**(单次,不是 task-by-task) |

**v6 关键变化**: Whole Branch Review 改用**最强模型一次性过**(而不是 task-by-task 全用 Opus 重审)

### 4.4 Plan 编写范式升级(视频第三刀:模型分层)

**Plan 新增必需结构块**:
- **Global Constraints block**: 版本下限、依赖上限、命名规则、所有任务绑定的不变条件,**原文复制**(确保执行方真的看到)
- **每任务的 Interfaces block**: 任务消费/产出什么(让只看自己任务的 Implementer 也清楚邻居的契约)
- **Right-sizing guidance**: 任务粒度控制 —— 太小浪费 review cycle,太大跳过 review

> **真实收益**: evals 测试中,这种 plan 写一次就过(control 需要 2-4 轮修),control 写过的版本还藏了个真 bug。

## 五、那些被实验"证明死了"的直觉(视频重点列的反直觉)

作者用 Fable 的 autoresearch loop 跑了 25 个实验(36 小时,$165 实际花费),有以下被明确"打死"的想法:

| 直觉 | 实验结论 | 数字 |
|---|---|---|
| **"给 Controller 加思考预算,反正思考贵"** | **反向爆炸** | turns 从 92 涨到 138,输出翻倍 |
| **"Plan 太长肯定浪费,加 PlanWordBudget"** | **测试信号 -62%** | 砍的内容包括测试/接口/任务结构 |
| **"用 Sonnet 生成 Plan 又便宜又不差"** | **任务结构崩溃** | 平均任务数从 Opus 的 5.8 → 3.6,5 个 plan 4 个任务过粗,reviewer 颗粒度没了 |
| **Narration Recipe 减少 reviewer 输出** | **稳定小收益** | -54% 输出,但每次只省 $0.1-0.3,不是主因 |
| **Conditional Implementer Tiering** | **有效** | 节省 $0.5-1/run,且 E22 证明它会**正确拒绝**用 Haiku 处理 prose plan |
| **Fixture-realism (Svelte)** | **有效** | 范围匹配 -24% |

**关键启示**:
1. **不是模型越贵/越cheap 越省钱**,而是**路由要对**
2. **Cap thinking 是错误方向** —— 前面少想,后面得多跑很多轮,省下来的思考 token 很快被修复轮次吃回去
3. **Plan 价值在它**后面的**执行 review 需要的矛点**(tests/interfaces/任务边界),砍掉这些后面每个 subagent 都得自己猜

## 六、上下文管理:Superpowers v6 的真正创新

视频核心论点:**贵的不是那几次推理,而是你让 AI 重复读同一批上下文**

### 6.1 多角色重复读同一批上下文的问题

```
Coordinator 看过 diff → Implementer 改过 → Reviewer 审查时又跑 git 重新曲变
                ↓              ↓                    ↓
            上下文窗口        输出                  输出
            (Observation)   (实现时)            (审查时,重新跑 git 命令)
```

**人类开发者这么做的成本**: 0(短期记忆 + IDE 缓存)
**LLM 这么做的成本**: 每一次输出都会变成 token,挂在上下文中,后面每一步继续付钱

### 6.2 v6 的解法:"证据带 + 任务交接"

> **类比**: 旧流程是让 Reviewer 自己进仓库翻资料,新流程是开工前把证据带装好,Reviewer 只读证据带。

| 信息 | v5 | v6 |
|---|---|---|
| Reviewer 看什么 | 自己跑 git 命令 | 读 review-package 文件 |
| Subagent 拿到什么 | Controller 把整份 Plan 塞给它 | 读自己的 task-brief 文件 |
| Plans 携带什么 | 散在各处 | Global Constraints + Interfaces 等结构化块 |
| Reviewer 看到的"全局" | grep/cat 拼凑 | task brief + diff package + whole-branch review |

## 九、铁律 / 反常识

1. **贵的不是那几次推理,而是让多 Agent 重复读同一批上下文** — 这才是 Superpowers v6 的核心洞察
2. **Plan 不能只求短** — 砍掉测试/接口/任务结构,后面每个 subagent 都得自己猜,成本反而升高
3. **Cap thinking 是错方向** — 前面少想,后面得多跑很多轮,省下来的思考 token 很快被修复轮次吃回去
4. **不要 reward GPU 的胆量,reward context 的精度** — reviewer 只看 diff 会偷偷重定义 spec 为 global constraints,要显式声明
5. **模型路由要看任务形态** — 不是越贵越好,机械任务便宜模型、判断密集任务贵模型,这是 v6 最大启发

## 十、落地清单(什么时候用 Superpowers)

**适合** ✅
- **多文件、跨模块功能开发**(Plan → TDD → Review → Whole-branch Review 流程收益大)
- **团队对 spec/code quality 一致性要求严格**(双 verdict 一次审查捕获)
- **项目规模 ≥ 中型**(代码量大,人类 review 跟不上的项目)
- **已有 Claude Code / Codex 等 IDE 账户**(Superpowers 是 plugin 层)

**不适合** ❌
- 一次性小脚本、不需要 spec 设计的项目
- 团队对脑暴/计划/设计流程没耐心(强 Plan-Driven 流程会让不耐心的用户卡住)
- 没有合法 API key 或无法启动 trusted-maintainer-only 模式(无法做 evals)

**5 分钟快速试**:
```bash
# Claude Code 官方 marketplace
/plugin install superpowers@claude-plugins-official

# 或 Superpowers marketplace
/plugin marketplace add obra/superpowers-marketplace
/plugin install superpowers@superpowers-marketplace
```
